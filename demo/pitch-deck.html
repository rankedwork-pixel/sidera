The Skill System

Skills are YAML files — version controlled, human-readable, and composable.

Three-Level Hierarchy

  • Department → shared context, vocabulary
  • Role → persona, principles, schedule
  • Skill → specific task definition

Context flows down: dept + role + skill compose into one prompt.

Safety Architecture

Write Operation Safety

  • Configurable safety caps — limit the scope of any single action
  • Double-execution prevention — executed_at field blocks re-runs
  • Previous values stored — every change is rollback-ready
  • Approval-gated — every write needs explicit human approval or matching auto-rule
  • Pre-action lesson check — blocks execution when past lessons contradict
  • Evidence snapshots — pre-action metrics captured for audit

Graduated Trust (Auto-Execute)

  • YAML rules per role define what can auto-execute
  • 10 condition operators (less_than, between, equals, etc.)
  • Constraints: daily limits, cooldown periods, platform restrictions
  • Global kill switch (default: OFF)
  • Skill proposals NEVER auto-execute — hard block
  • Human steward accountable for each agent's behavior

Information Security

Four clearance levels: PUBLIC INTERNAL CONFIDENTIAL RESTRICTED — Skills gated by minimum clearance. Roles have clearance levels. Inter-agent data filtered by clearance.

Conversational Mode

Every role is dual-mode: autonomous (scheduled briefings) and conversational (Slack threads).

How It Works

  • @Sidera talk to the [role name] starts a thread
  • RoleRouter identifies the role (regex + semantic fallback)
  • Thread pinned to one role via DB (thread_ts → role_id)
  • Each reply triggers a new agent turn with full tool access
  • Automatic memory extraction after every turn

Write Ops in Conversations

  • System extracts them, posts Approve/Reject buttons in-thread
  • On approval, executes via connector and posts result

Memory System

Tiered Architecture

  • Hot memories (≤ 90 days) — auto-injected into every role run, capped at 2000 tokens
  • Cold archive (> 90 days) — never deleted, searchable via MCP tool
  • Weekly consolidation — merges similar memories, prevents bloat
  • Memory index — when >20 memories, compact index with on-demand loading
  • Memory versioning — supersedes_id chains track evolution

Memory Types (8)

Decision Anomaly Pattern Insight Lesson Commitment Relationship Steward Note

Steward notes: highest priority, injected first, only humans can create. Decisions from approvals. Lessons from reflection. Commitments from conversations.

Post-Run Reflection + Lesson-Principle Linking

After every role run, the agent asks: "What was hard? What data was missing?" Answers saved as lesson memories. Each lesson linked to the role's decision-making principles. Compounding value — agents learn from every single run.

Self-Improving Agents

Skill Evolution

Agents propose changes to their own skill definitions via MCP tool. Same approval pipeline as any other action:

Agent proposes → Approval queue → Human reviews diff → Approve/Reject → Written to DB

Graduated evidence thresholds: 3 lessons → guidance only, 5 → +supplements, 7 → +prompts/model.

FORBIDDEN FIELDS: requires_approval, manages, is_active

Role Evolution

Department heads can propose new roles or modify existing ones:

  • Only managers can propose
  • Department-scoped — can't create roles outside their dept
  • New roles auto-added to manager's manages list
  • NEVER auto-executes

Reflection → skill evolution pipeline: recurring friction (3+ lessons) auto-generates improvement proposals.

Dynamic Org Chart

YAML on disk is seed data. Database is the override layer. Slack commands + REST API for runtime management. All mutations audit-logged. Soft delete. Graceful DB fallback to disk-only.

Why Inngest, Not Celery?

Durable by default — steps are checkpointed and retried. No broker to manage. Async human approval is a first-class pattern (pause workflow, resume on event). Dead letter queue for failures. Visual dashboard for debugging. Serverless-compatible.

Autonomous Behaviors

Proactive Heartbeats

Roles wake up on configurable cron schedules and freely investigate their domain.

  • Configurable per role (every 15 min, hourly, business hours only, etc.)
  • Open-ended investigation — no pre-defined task
  • Findings posted to Slack automatically
  • Cooldown prevents rapid-fire runs
  • Global kill switch for safety

Peer-to-Peer Messaging

Roles communicate asynchronously without manager mediation.

  • MCP tools: send_message_to_role, check_inbox, reply_to_message
  • Messages injected into context on next run
  • Anti-loop: max 3 messages/run, max 5 chain depth
  • Self-messaging blocked
  • Stale message expiry via data retention workflow

The Big Picture

Sidera is not a chatbot. It's a framework for building AI workforces — teams of specialized agents with roles, memory, principles, and safety controls, orchestrated through durable workflows and gated by human approval.

The accumulated expertise in each skill — the context, examples, guidelines, and learned memories —
is the product. The framework just runs it.
