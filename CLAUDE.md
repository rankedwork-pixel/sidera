# Sidera ‚Äî AI Agent Framework

## What This Is
A framework for building **AI employees** ‚Äî autonomous agents that connect to external APIs, analyze data on a schedule, recommend actions via Slack, and execute with human approval.

The core pattern is domain-agnostic: connect data sources ‚Üí teach skills via YAML ‚Üí get structured Slack briefings with approve/reject buttons ‚Üí execute approved actions ‚Üí log everything. Swap connectors and skills for any domain.

**Target users:** Developers and operators building AI-powered automation for any domain.

## Architecture Decisions
- **Agent Framework:** Direct Anthropic API (Python SDK) with MCP-style tool registry
- **Models:** Haiku (parsing + data collection) ‚Üí Sonnet (analysis) ‚Üí Opus (complex strategy). Daily briefings use three-phase routing: Haiku pulls data (~$0.02), Sonnet analyzes (~$0.15), Opus adds strategic layer (~$0.35) ‚Äî total ~$0.52 vs ~$1.50-3.00 single-model. **Extended thinking:** All Sonnet/Opus calls use `thinking_budget=10000` tokens for deep internal reasoning. Interleaved thinking enables reasoning between tool calls. Config: `extended_thinking_enabled` (default ON), `extended_thinking_budget_tokens` (default 10000). **Hybrid routing:** Provider abstraction layer (`src/llm/`) routes structured tasks (routing, extraction, reflection) to cheaper OpenAI-compatible providers when `external_llm_enabled=True`. Quality-critical tasks (analysis, conversation, strategy, tool use) always stay on Claude. Automatic fallback: external failure ‚Üí transparent Claude retry. `TaskType` enum on every LLM call enables per-task-type cost tracking and routing decisions.
- **Skill System:** Three-level hierarchy: Department ‚Üí Role ‚Üí Skill. YAML skills on disk organized into `library/<dept>/<role>/<skill>` structure. SkillRegistry discovers departments (`_department.yaml`), roles (`_role.yaml`), and skills (flat or folder-based). Context flows down: department.context ‚Üí role.persona ‚Üí skill.system_supplement. RoleExecutor runs all briefing_skills for a role; DepartmentExecutor runs all roles; ManagerExecutor handles manager roles (delegation + synthesis). SkillRouter (Haiku) handles semantic matching for free-text queries.
- **Manager Roles:** A manager is a `RoleDefinition` with a `manages` field ‚Äî no new DB tables or abstractions. Four-phase pipeline: (1) run own briefing_skills, (2) LLM delegation decision (Sonnet/Haiku, no tools, max_turns=1), (3) sub-role execution (sequential, each checkpointed), (4) LLM synthesis (Sonnet, no tools, max_turns=1). Delegation failure ‚Üí activate ALL sub-roles (safe fallback). Sub-roles run inline in the manager workflow (not separate Inngest events) so results stay in scope for synthesis. Recursive managers supported with depth limit (max 3).
- **Conversational Mode:** Roles are dual-mode: autonomous (scheduled) + conversational (Slack threads). User @mentions Sidera or uses `/sidera chat <role>` ‚Üí RoleRouter identifies role (regex + Haiku semantic fallback) ‚Üí thread pinned to that role via `conversation_threads` DB table ‚Üí each reply triggers `conversation_turn_workflow` Inngest event ‚Üí agent responds in-character with full tool access. Stateless per-turn: thread history fetched from Slack API each turn. Limits: 20 turns, 24h timeout, $5/thread. Cost: ~$0.03-0.15/turn (Sonnet). **Write operations in conversations:** Agent generates JSON `{"recommendations": [...]}` blocks ‚Üí `_extract_recommendations()` strips from visible response ‚Üí `_process_recommendations_inline()` creates DB approvals + posts Approve/Reject buttons in-thread ‚Üí `_execute_approved_action_inline()` runs on approval and posts result. Dev-mode inline runner (`_run_conversation_turn_inline`) bypasses Inngest for local testing.
- **Dynamic Org Chart:** YAML on disk is the baseline/seed data. DB entries with the same ID **replace** disk entries entirely; new IDs are **added**. `load_registry_with_db()` is the standard entry point: loads disk via `SkillRegistry.load_all()`, then overlays DB definitions via `merge_db_definitions()`. DB unavailable ‚Üí disk-only fallback (silent, logged). Managed via `/sidera org` Slack commands (8 subcommands) and REST API (16 endpoints on `/api/org/`). All changes audit-logged. Soft delete (is_active=False) ‚Äî inactive entries not loaded into registry.
- **Skill Evolution:** Agents can propose modifications to skill definitions (or propose entirely new skills) via `propose_skill_change` MCP tool. Treated as a write operation ‚Äî same pipeline as budget changes. Agent proposes ‚Üí approval queue ‚Üí human reviews diff in Slack ‚Üí approve/reject ‚Üí on approval, written to DB via Dynamic Org Chart. FORBIDDEN_FIELDS (`requires_approval`, `manages`, `is_active`) prevent agents from bypassing their own safety controls. Hard block in `should_auto_execute()` ensures skill proposals ALWAYS require human review. Proposals collected via module-level `_pending_proposals` list, merged into recommendations after agent turn.
- **Listen-Only Meetings:** Recall.ai bot joins meetings via URL, captures transcript via webhooks. Listen-only ‚Äî no voice output. MeetingSessionManager orchestrates transcript capture. Post-call: meeting_end_workflow summarizes transcript, extracts action items, triggers manager delegation with `meeting_context`. MeetingSession DB table tracks status (joining ‚Üí in_call ‚Üí ended ‚Üí delegating ‚Üí completed). Slack `/sidera meeting join|status|leave` for control. Limits: 2h max, $10/session cost cap.
- **Orchestration:** Inngest durable functions ‚Äî scheduled cron trigger, checkpointed steps, async human approval
- **Database:** PostgreSQL via Supabase (accounts, campaigns, metrics, audit_log, approval_queue)
- **Cache:** Redis via Upstash (API response caching, session state, briefing result cache with 2h TTL ‚Äî never sole source of truth)
- **Notifications:** Slack Bolt SDK (interactive approve/reject buttons)
- **Deploy:** Railway (cron + background workers)
- **Frontend:** Streamlit (MVP) ‚Üí Next.js later

## Development Workflow (IMPORTANT ‚Äî follow after every feature/fix)

After completing any code changes (new features, bug fixes, refactors), ALWAYS run the cleanup pipeline before considering the work done:

1. **Lint + format:** `make lint` (or `python3 -m ruff check src/ tests/ dashboard/`)
2. **Run tests:** `make test` (or `python3 -m pytest tests/ -v --tb=short -q`)
3. **Sync docs:** `make sync-docs` (or `python3 scripts/doc_sync.py`)
   - If counts are stale, run `make update-docs` to auto-fix them
4. **Fix any issues** found in steps 1-3 before stopping

Or use the single command: **`make cleanup`** (runs format + lint + test + sync-docs).

The `scripts/doc_sync.py` script counts 11 codebase metrics (tests, DB methods, workflows, migrations, MCP tools, connectors, skills, roles, departments, Slack methods, test files) and compares them against CLAUDE.md, README.md, and Instructions. If you add a new DB method, MCP tool, workflow, migration, connector, or skill, the doc sync will catch the stale count.

**Instructions file:** The `Instructions` file at the project root is a comprehensive rebuild document (5000+ lines). When shipping new features, update the relevant sections in `Instructions` ‚Äî schema changes, new MCP tools, new YAML fields, new migrations, new DB methods, etc. The doc_sync script tracks migration and DB method counts in this file automatically.

## Key Design Principles
1. **Read-first, write-gated:** Agent never modifies external systems without human approval (Tier 3), or auto-executes only when matching pre-approved rules (Tier 2, default OFF)
2. **Stateless agent, stateful database:** Each run is a fresh Claude conversation loading context from PostgreSQL
3. **Verify every claim:** All metrics cross-checked against raw API data before presenting
4. **Circuit breakers:** Max 20 tool calls per cycle, max $10 LLM cost per account per day
5. **Domain-agnostic core:** The agent loop, approval flow, audit trail, and skill system work for any domain. Only connectors and skills are domain-specific.
6. **Skills are the product:** The framework runs skills. The accumulated expertise in each skill (examples, context, guidelines, rubrics) is what makes the agent good at its job.

5. **Agent Stewardship ‚Äî Human Accountability at Scale** ‚Äî Every AI role (and optionally every department) has a designated **steward**: a real human who is personally accountable for that agent's behavior, outputs, skill definitions, and evolution. This is the "who gets the credit / who takes the blame" layer that makes AI agents manageable at organizational scale. Without it, agents become orphans ‚Äî nobody owns the outcomes, nobody curates the skills, nobody answers when something goes wrong.

   **Core concepts:**

   - **Steward** ‚Äî the human currently responsible for an AI role's performance and behavior. One human can steward multiple roles. One role has exactly one steward (single point of accountability). Think: "manager of the AI employee."
   - **Creator** ‚Äî the human who originally built or configured a role/skill. Tracked separately from steward because the person who built it may not be the person running it day-to-day. Preserved permanently for credit attribution.
   - **Stewardship scope** ‚Äî stewards can be assigned at the role level (granular) or department level (all roles in that department). Role-level assignment overrides department-level. A department steward is the default for any un-assigned role in their department.
   - **Accountability chain** ‚Äî for manager roles that delegate to sub-roles: the manager's steward is accountable for delegation decisions, but each sub-role's steward is accountable for that sub-role's execution. Escalation flows up: sub-role steward ‚Üí manager steward ‚Üí department steward.

   **What stewardship means concretely:**

   - **Approval routing** ‚Äî when an AI role generates a recommendation requiring approval, it routes to the steward first (DM or steward's channel), not just a shared channel. Other approver+ users can still approve, but the steward is the primary responsible party and gets notified.
   - **Skill evolution gating** ‚Äî when an agent proposes a skill change or new skill, only the steward of that role (or their department steward) can approve the proposal. Prevents random approvers from modifying agents they don't understand.
   - **Escalation target** ‚Äî when an agent hits errors, exceeds cost thresholds, or gets rejected repeatedly, the steward is the escalation point. System sends them a "your agent needs attention" alert.
   - **Performance dashboard** ‚Äî each steward gets a view of their agents' KPIs: approval rate, cost per run, error rate, memory growth, skill evolution history, auto-execute hit rate. This is the "manager reviewing their employee's performance" view.
   - **Memory curation** ‚Äî stewards can inject `steward_note` memories into their agents' memory (guidance, corrections, context the agent wouldn't discover on its own) and can archive/override agent-generated memories that are wrong or stale.
   - **Principles ownership** ‚Äî the `principles` field on roles (decision-making heuristics) is owned by the steward. When the agent proposes a principle change via skill evolution, the steward approves. The steward can also directly edit principles via Slack or API.
   - **Audit attribution** ‚Äî every audit log entry records which steward was responsible at the time. If stewardship transfers, the audit trail shows who was steward for which period.
   - **Handoff protocol** ‚Äî stewardship can transfer between humans (like on-call rotation). Transfer requires: (a) outgoing steward or admin initiates, (b) incoming steward accepts, (c) system generates a stewardship briefing (role summary, recent performance, open issues, memory highlights) for the incoming steward. Full audit trail of all transfers.

   **Data model changes:**

   - **`stewardship` join table** (new) ‚Äî `id`, `user_id` (FK to users), `scope_type` (role | department), `scope_id` (role_id or dept_id), `is_creator` (boolean ‚Äî was this person the original creator?), `assigned_at`, `assigned_by`, `released_at`, `released_by`, `is_active`. Supports history: when stewardship transfers, old row gets `released_at` set, new row created. Full timeline of who owned what and when.
   - **`org_roles` / `org_departments`** ‚Äî add `steward_user_id` (String 255, nullable) as a denormalized fast-lookup field (mirrors the active stewardship row). Updated on stewardship assignment/transfer.
   - **`RoleDefinition` / `DepartmentDefinition`** ‚Äî add `steward` field (optional string). Populated from DB overlay or YAML (YAML is for initial seed; DB is the live truth).
   - **`audit_log`** ‚Äî add `steward_user_id` (String 255, nullable) column. Populated at write time from the role/department's current steward. Immutable snapshot (doesn't change if stewardship later transfers).
   - **`approval_queue`** ‚Äî add `steward_user_id` (String 255, nullable). Approval routing logic checks this field to notify the right human.
   - **`role_memory`** ‚Äî add `STEWARD_NOTE` memory type. These memories are injected with highest priority (above agent-generated memories) and cannot be overridden or archived by the agent. Only the steward (or admin) can modify steward notes.

   **Slack interface:**

   - `/sidera steward list` ‚Äî show all stewardship assignments (who owns what)
   - `/sidera steward show <role_id>` ‚Äî show steward for a specific role, plus history
   - `/sidera steward assign <role_id> <@user>` ‚Äî assign steward (admin or current steward)
   - `/sidera steward transfer <role_id> <@user>` ‚Äî transfer with handoff briefing
   - `/sidera steward release <role_id>` ‚Äî release stewardship (leaves role unassigned, alerts admins)
   - `/sidera steward dashboard` ‚Äî steward's personal view of all their agents' performance
   - `/sidera steward note <role_id> <text>` ‚Äî inject a steward note into role memory
   - Approval messages include "Steward: @username" label so everyone knows who's responsible
   - Escalation DMs: "Your agent [role_name] has failed 3 consecutive runs. Here's what happened: ..."

   **REST API:**

   - `GET /api/stewardship/` ‚Äî list all assignments
   - `GET /api/stewardship/{role_id}` ‚Äî get steward for role
   - `GET /api/stewardship/user/{user_id}` ‚Äî get all roles a user stewards
   - `POST /api/stewardship/assign` ‚Äî assign steward

   - `DELETE /api/stewardship/{role_id}` ‚Äî release stewardship
   - `GET /api/stewardship/{role_id}/history` ‚Äî full stewardship timeline
   - `GET /api/stewardship/dashboard/{user_id}` ‚Äî steward performance dashboard data

   **Safety constraints:**

   - Agents CANNOT modify their own steward (add `steward_user_id` to `FORBIDDEN_FIELDS` in skill/role evolution)
   - Agents CANNOT create or modify steward_note memories
   - Unassigned roles generate weekly "orphan agent" alerts to admins
   - Stewardship is required before a role can be activated in production (configurable via `stewardship_required` global setting)
   - Auto-execute rules can only be created/modified by the role's steward (not any admin)

   **Implementation leverages existing infrastructure:**

   - RBAC system (users table, permission checks) ‚Äî extend with stewardship-aware permissions
   - Org chart (DB overlay, REST API, Slack commands) ‚Äî follow same patterns for stewardship CRUD
   - Approval pipeline (routing, `decided_by`) ‚Äî add steward-first routing
   - Audit log (event tracking) ‚Äî add `steward_user_id` column
   - Memory system (memory types, injection) ‚Äî add `STEWARD_NOTE` type
   - Skill evolution (forbidden fields, approval gating) ‚Äî add steward-scoped approval
   - Alembic migrations ‚Äî one migration for new table + column additions

   **Why this matters at scale:** When an organization has 50+ AI roles across 10 departments, the question "who is responsible for what the marketing analyst agent did yesterday?" must have a clear, instant answer. Stewardship provides: (a) clear accountability chains, (b) informed approval routing, (c) expertise concentration (the steward deeply understands their agents), (d) quality incentives (steward's reputation tied to agent performance), (e) safe handoffs when people change roles, and (f) an audit trail that satisfies compliance requirements. It's the difference between "we have AI agents" and "we have AI agents that are managed responsibly."
