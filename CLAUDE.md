# Sidera — AI Agent Framework

## What This Is
A framework for building **AI employees** — autonomous agents that connect to external APIs, analyze data on a schedule, recommend actions via Slack, and execute with human approval.

The core pattern is domain-agnostic: connect data sources → teach skills via YAML → get structured Slack briefings with approve/reject buttons → execute approved actions → log everything. Swap connectors and skills for any domain.

**Target users:** Developers and operators building AI-powered automation for any domain.

## Architecture Decisions
- **Agent Framework:** Direct Anthropic API (Python SDK) with MCP-style tool registry
- **Models:** Haiku (parsing + data collection) → Sonnet (analysis) → Opus (complex strategy). Daily briefings use three-phase routing: Haiku pulls data (~$0.02), Sonnet analyzes (~$0.15), Opus adds strategic layer (~$0.35) — total ~$0.52 vs ~$1.50-3.00 single-model. **Extended thinking:** All Sonnet/Opus calls use `thinking_budget=10000` tokens for deep internal reasoning. Interleaved thinking enables reasoning between tool calls. Config: `extended_thinking_enabled` (default ON), `extended_thinking_budget_tokens` (default 10000). **Hybrid routing:** Provider abstraction layer (`src/llm/`) routes structured tasks (routing, extraction, reflection) to cheaper OpenAI-compatible providers when `external_llm_enabled=True`. Quality-critical tasks (analysis, conversation, strategy, tool use) always stay on Claude. Automatic fallback: external failure → transparent Claude retry. `TaskType` enum on every LLM call enables per-task-type cost tracking and routing decisions.
- **Skill System:** Three-level hierarchy: Department → Role → Skill. YAML skills on disk organized into `library/<dept>/<role>/<skill>` structure. SkillRegistry discovers departments (`_department.yaml`), roles (`_role.yaml`), and skills (flat or folder-based). Context flows down: department.context → role.persona → skill.system_supplement. RoleExecutor runs all briefing_skills for a role; DepartmentExecutor runs all roles; ManagerExecutor handles manager roles (delegation + synthesis). SkillRouter (Haiku) handles semantic matching for free-text queries.
- **Manager Roles:** A manager is a `RoleDefinition` with a `manages` field — no new DB tables or abstractions. Four-phase pipeline: (1) run own briefing_skills, (2) LLM delegation decision (Sonnet/Haiku, no tools, max_turns=1), (3) sub-role execution (sequential, each checkpointed), (4) LLM synthesis (Sonnet, no tools, max_turns=1). Delegation failure → activate ALL sub-roles (safe fallback). Sub-roles run inline in the manager workflow (not separate Inngest events) so results stay in scope for synthesis. Recursive managers supported with depth limit (max 3).
- **Conversational Mode:** Roles are dual-mode: autonomous (scheduled) + conversational (Slack threads). User @mentions Sidera or uses `/sidera chat <role>` → RoleRouter identifies role (regex + Haiku semantic fallback) → thread pinned to that role via `conversation_threads` DB table → each reply triggers `conversation_turn_workflow` Inngest event → agent responds in-character with full tool access. Stateless per-turn: thread history fetched from Slack API each turn. Limits: 20 turns, 24h timeout, $5/thread. Cost: ~$0.03-0.15/turn (Sonnet). **Write operations in conversations:** Agent generates JSON `{"recommendations": [...]}` blocks → `_extract_recommendations()` strips from visible response → `_process_recommendations_inline()` creates DB approvals + posts Approve/Reject buttons in-thread → `_execute_approved_action_inline()` runs on approval and posts result. Dev-mode inline runner (`_run_conversation_turn_inline`) bypasses Inngest for local testing.
- **Dynamic Org Chart:** YAML on disk is the baseline/seed data. DB entries with the same ID **replace** disk entries entirely; new IDs are **added**. `load_registry_with_db()` is the standard entry point: loads disk via `SkillRegistry.load_all()`, then overlays DB definitions via `merge_db_definitions()`. DB unavailable → disk-only fallback (silent, logged). Managed via `/sidera org` Slack commands (8 subcommands) and REST API (16 endpoints on `/api/org/`). All changes audit-logged. Soft delete (is_active=False) — inactive entries not loaded into registry.
- **Skill Evolution:** Agents can propose modifications to skill definitions (or propose entirely new skills) via `propose_skill_change` MCP tool. Treated as a write operation — same pipeline as budget changes. Agent proposes → approval queue → human reviews diff in Slack → approve/reject → on approval, written to DB via Dynamic Org Chart. FORBIDDEN_FIELDS (`requires_approval`, `manages`, `is_active`) prevent agents from bypassing their own safety controls. Hard block in `should_auto_execute()` ensures skill proposals ALWAYS require human review. Proposals collected via module-level `_pending_proposals` list, merged into recommendations after agent turn.
- **Orchestration:** Inngest durable functions — scheduled cron trigger, checkpointed steps, async human approval
- **Database:** PostgreSQL via Supabase (accounts, campaigns, metrics, audit_log, approval_queue)
- **Cache:** Redis via Upstash (API response caching, session state, briefing result cache with 2h TTL — never sole source of truth)
- **Notifications:** Slack Bolt SDK (interactive approve/reject buttons)
- **Deploy:** Railway (cron + background workers)
- **Frontend:** Streamlit (MVP) → Next.js later

## Key Design Principles
1. **Read-first, write-gated:** Agent never modifies external systems without human approval (Tier 3), or auto-executes only when matching pre-approved rules (Tier 2, default OFF)
2. **Stateless agent, stateful database:** Each run is a fresh Claude conversation loading context from PostgreSQL
3. **Verify every claim:** All metrics cross-checked against raw API data before presenting
4. **Circuit breakers:** Max 20 tool calls per cycle, max $10 LLM cost per account per day
5. **Domain-agnostic core:** The agent loop, approval flow, audit trail, and skill system work for any domain. Only connectors and skills are domain-specific.
6. **Skills are the product:** The framework runs skills. The accumulated expertise in each skill (examples, context, guidelines, rubrics) is what makes the agent good at its job.
